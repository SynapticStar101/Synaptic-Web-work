Barney's BinDaRoids: Build Journal

From "I have an idea" to a live game on the internet. Every problem, every fix, and the reason why things needed changing — in plain English.

This is the behind-the-scenes story of building Barney's BinDaRoids — a custom browser game playable on any phone, tablet or computer, with a live global leaderboard. No app store. No downloads. Just a link.

It was built entirely using AI (Claude Code), with a non-developer directing the decisions. Every problem that came up, every fix applied, and the reason why things needed changing is documented here — in plain English.

Entry 1 — The Brief

What we wanted: a web-based arcade game based on the classic Asteroids formula, but with personality. The ship would be Barney's bin wagon. The asteroids would be household rubbish — wheelie bins, recycling bins, black bags, old tellies. A bonus enemy rat ship would fly across occasionally for big points. Retro arcade visuals, chiptune music, sound effects, increasing difficulty per level, and it had to work on mobile, tablet and desktop without any installation.

Why a browser game? Browser games run on absolutely any device with a web browser. No app store approval. No "only works on Android." You send someone a link and they're playing in seconds. For something fun and shareable, it's the ideal format.

The core technology chosen:

  • A single HTML file containing everything — the visuals, the sounds, the game logic, all in one
  • Canvas-based drawing (the game draws every single frame itself, like a flipbook, 60 times per second)
  • Web Audio API for synthesised chiptune music and sound effects — no audio files needed, all generated mathematically on the fly
  • Everything self-contained so it could be hosted anywhere for free

Entry 2 — Getting the Game Running

What was built first: the complete game in one session — the bin wagon ship (drawn programmatically as a green truck with Barney's face), five types of rubbish asteroid, the rat ship bonus enemy, particle explosion effects, floating score popups, a starfield background, chiptune music, and all the sound effects.

Why draw everything in code rather than use images? Images would need to be hosted separately, loaded at the right time, and managed as files. Code-drawn graphics are instant, scale to any screen size perfectly, and mean the entire game is genuinely one file. The retro look also suits the arcade aesthetic.

Why synthesise the music in code? Same reason — no audio files to host or load. The Web Audio API can generate waveforms mathematically. Square waves for the melody (classic 8-bit sound), sawtooth waves for the bass, noise bursts for explosions and thrust. Zero loading time.

An interesting design constraint: the ship had to feel like a bin wagon — heavy, a bit slow to react, satisfying when it finally gets moving. Early physics used acceleration that built up over time, which created that tank-like feel.

Entry 3 — The Play Again Button Broke

The problem: after a game over, the Play Again button stopped working. Clicking it did nothing.

Why it broke — explained simply: imagine a timer that counts down to zero, and when it hits zero it's supposed to show a "Game Over" screen once. The bug was that the timer hit zero, showed the screen, but the condition checking "has the timer hit zero?" was still true every frame after that. So every frame (60 times a second), the game was rebuilding the Game Over screen from scratch — including the button — destroying the button that had just been set up a fraction of a second earlier. By the time your finger reached to tap it, it had already been replaced thousands of times.

The fix: before calling the "show game over screen" function, immediately flip the game's internal state to a new mode called "over" — a final resting state that the timer condition never checks again. The screen gets built exactly once, the button stays intact, and clicking it works.

💡
The lesson: game loops run constantly. Any code that should run once needs an explicit "I've already done this" flag, otherwise it runs every frame forever.

Entry 4 — Adding a High Score Leaderboard

What we wanted: a global leaderboard — not just local scores saved on your own device, but a real shared table where anyone in the world who plays can appear. After game over, you type your name, it saves to a database, and you see the top 10 alongside yours.

The technology chosen — Supabase: Supabase is a free online database service. Think of it like a spreadsheet in the cloud that your game can read from and write to. The game sends your name, score, and level to Supabase after every game over. When the leaderboard loads, it asks Supabase for the top 10 scores, sorted highest first.

The security setup required: databases need rules about who can do what. For this game:

  • Anyone can read the scores (so the leaderboard is public)
  • Anyone can write a new score (so players can submit)
  • Nobody can delete or edit other people's scores

These rules are called Row Level Security policies. Without them, the database rejects all requests by default — a sensible "deny everything until you explicitly allow it" approach.

Why Supabase rather than something else? It has a generous free tier, a clean interface for setting up tables, and a JavaScript library that the game can load directly from the internet without any installation. The whole integration is about 20 lines of code.

Entry 5 — GitHub Pages: The Publishing Problem

What we wanted: publish the game to a public URL so anyone with the link can play it. GitHub Pages is a free hosting service that turns a code repository into a live website.

The problem that appeared: after enabling GitHub Pages in the settings, the button was greyed out with a message: "Upgrade or make this repository public to enable Pages."

Why it happened: GitHub's free tier only allows GitHub Pages on public repositories. The game's code repository was set to private (hidden from the internet), which blocked the hosting service entirely.

The decision: make the repository public. This meant anyone could see the source code of the game — but for a fun browser game with no sensitive data, that's completely fine. The trade-off: free hosting forever vs. keeping code private (which would require a paid plan).

What "public repository" actually means: the code is visible if someone specifically goes looking for it. The game's URL stays the same. Players just click the link and play — they never see the code. It's the same as how most open-source software works.

Entry 6 — The Leaderboard That Wouldn't Show

The problem: even after Supabase was configured with real credentials, the leaderboard screen would either show a "Connect Supabase" message (wrong — it was connected) or go completely blank after submitting a name.

Two separate bugs were causing this.

Bug 1 — Wrong order of checks: the code was doing things in this order:

  1. Show the leaderboard screen (with a "Loading..." message)
  2. Go off to the internet to fetch scores (this takes a moment)
  3. While waiting, check whether Supabase is even connected

Step 3 should have happened before step 2. By the time the check ran, the internet fetch had already started. If the connection check said "not connected," the code tried to update a "Loading..." text on screen — but by then, the screen layout might have shifted and that piece of text no longer existed where the code expected it. The result: a silent crash, nothing displayed.

The fix: move the connection check to before the internet fetch. If not connected, immediately show the local score. If connected, then go fetch scores.

Bug 2 — The Supabase library not loading: the Supabase JavaScript code is loaded from an external CDN (a delivery network on the internet). If that CDN was slow or the browser blocked it, the game would start without Supabase available at all — and silently fall back to "not connected" mode with no error shown to the player.

The fix for reliability: add null safety checks throughout — any time the code tries to update the screen after an internet operation, it first confirms the screen element still exists before touching it.

Entry 7 — Mobile Controls: The D-Pad Problem

The original mobile controls: five circular buttons arranged like a cross — left, right, up (thrust), down (brake), and a fire button. Functional, but awkward. Your thumb had to hit precise small targets while also watching the screen. Changing direction meant lifting your thumb, repositioning, and pressing a different button.

Why this was genuinely hard to use: the original Asteroids cabinet had a physical joystick with tactile feedback — you could feel which way you were pushing without looking. On a touchscreen, there's no physical reference point. Your thumb drifts. You think you're pressing "thrust" but you've drifted onto "right." The game gets frustrating.

The solution — a virtual joystick: replace the five buttons with a single circle. You press anywhere inside it and drag in the direction you want to go. Push up-left to rotate left while thrusting. Push down to brake. The position of your thumb within the circle is read continuously, and the game responds proportionally.

Why this works better: your thumb always stays in the same zone — the joystick base. There's no need to lift and reposition. Diagonal inputs (thrust while turning) happen naturally. The game becomes much more about the action on screen and much less about finding the right button.

The fire button was kept as a separate button on the right side, because shooting is a deliberate separate action and benefits from a dedicated tap.

How the joystick knows what you're doing: the game tracks the distance and direction from the joystick's centre to where your thumb currently is. Past a small dead zone (to ignore accidental micro-movements), it translates the position into left/right/up/down signals — the same signals the keyboard arrow keys send on desktop.

Entry 8 — Ship Speed: From Physics Simulation to Feel

The original ship physics: classic Asteroids uses real momentum simulation. Press thrust and you add to your current speed in that direction. Release thrust and you drift. In the original game, this is the whole point — managing your drift is the skill.

Why it wasn't working well here: on a small touch screen with a virtual joystick, managing momentum adds a layer of difficulty that fights the controls rather than complementing them. You'd thrust to avoid a bin, release the joystick, and continue drifting toward another bin that appeared from a different direction. The ship felt out of control.

The change: switch from accumulating speed to constant speed. When thrust is active, the ship moves at exactly 4 pixels per frame in the direction it's facing — no more, no less. When thrust is released, it coasts briefly (the existing speed fades quickly) and then stops. There's no momentum to manage.

What this changes about the game feel: the ship becomes more predictable. You press toward something and it goes there. You release and it stops shortly after. The skill shifts from physics management to navigation and aim — which works better on a touch screen.

The brake button still exists and now acts as an emergency stop (faster fade), useful when you're about to overshoot into a bin.

Entry 9 — Mobile Viewport: Portrait and Landscape Inconsistency

The problem: on mobile phones, switching between portrait (holding the phone upright) and landscape (holding it sideways) should resize the game to fill the new screen shape. Sometimes it worked. Sometimes the game canvas stayed the old size, leaving black bars or cut-off edges. On iPhones specifically, the game could be the wrong height because of the browser's address bar eating into the screen space.

Why this happens — explained simply: a phone screen's "available space" is not simply its physical dimensions. The browser's address bar, the iOS home indicator bar at the bottom, and the notch cut-out on newer iPhones all take chunks out of the usable area. The web platform has evolved different ways to express "how much screen is actually available" and they don't always agree:

  • 100vh — 100% of the viewport height. On iOS Safari, this can include space behind the address bar that isn't actually visible. Classic source of content being cut off at the bottom.
  • 100svh — "small viewport height" — always the smallest guaranteed visible area, with all browser chrome showing. This is what we switched to for the body element, so the game never gets taller than what's actually shown.
  • visualViewport API — a more modern JavaScript interface that reports the true visible rectangle in real time, updating as the keyboard appears, as orientation changes, as the browser chrome hides/shows. The resize function now uses this if available.

The orientation change fix: when a phone rotates, the browser fires an orientationchange event. The game listens for this and triggers a resize — but with a 120 millisecond delay. Why the delay? The browser doesn't instantly finish the rotation animation. If you resize the canvas the moment the event fires, you measure the old dimensions mid-rotation. Waiting 120ms lets the browser settle before measuring.

The notch fix (viewport-fit=cover): on iPhones with a notch (the cut-out at the top) and a home bar (the swipe indicator at the bottom), the game's touch controls — the joystick and fire button — could be hidden behind these physical features when holding the phone sideways. Adding viewport-fit=cover to the page settings tells the browser to extend content behind the notch, and then using safe area inset values in CSS pushes the controls just far enough inward to clear the notch and home bar.

Entry 10 — Version Numbers: Knowing What's Live

The problem that prompted this: after making changes and pushing them to GitHub, it wasn't clear whether the live game on GitHub Pages was the old version or the new version. GitHub Pages has a deployment delay of 1–2 minutes, and browser caching can make old versions stick around even after the new one is live.

The fix: add a version number (v1.1) to the main menu screen. From now on, every time changes are deployed, the version number increments (v1.2, v1.3, etc.). If the number on screen matches what was just deployed, the latest version is live. If it still shows an old number, either GitHub Pages hasn't finished deploying yet, or the browser has cached the old page — a hard refresh (Ctrl+Shift+R on desktop, or close and reopen on mobile) will clear it.

💡
Why version numbers matter: without them, you're guessing. With them, you have a single visible confirmation that the correct code is running. It takes five seconds to add and saves minutes of confusion every time something is deployed.

Things That Went Well From the Start

  • Single-file architecture — having everything in one file made the whole project easy to move, share, deploy, and understand. No build process, no dependencies to install, no configuration files.
  • Canvas-based rendering — drawing everything in code meant no asset management, instant loading, and perfect scaling to any screen.
  • Synthesised audio — no audio files to host, no loading delays, and the chiptune aesthetic was built in from the start rather than bolted on.
  • Supabase free tier — zero cost, generous limits for a game with this scale, and the JavaScript library just works with a link.

What This Project Demonstrates

You don't need to be a developer to ship a working game. You need:

  1. A clear brief (what should it look, feel, and do?)
  2. A feedback loop (play it, find what's wrong, describe it plainly)
  3. Persistence when something breaks (every bug here had a logical cause and a fix)
  4. A system for knowing when your fix is live (version numbers)

The technology changes rapidly. The process of describing what you want, testing it, and iterating doesn't.

— Barney, Non-Dev, Accidental Asteroid Blaster
