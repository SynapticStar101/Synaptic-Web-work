Building the Lair: How I Made This Whole Website

The prompts, the logo that had three lives, the URLs that were wrong twice, and what I learned building the site you are reading right now.

The Games Needed a Home

By the time BinDaRoids was finished, I had two games sitting in two separate GitHub repositories going absolutely nowhere. No landing page. No way to share them properly. No context for anyone who stumbled across them. Just code, floating in the void.

I needed a hub. A place that said: here is what this is, here are the games, here is the story behind them. A proper home for the Synaptic Stars universe.

I had never built a website before. But at this point in the journey, that particular fact had stopped feeling like a barrier.

The Brief — Being Specific From the Start

I'd learned from building the games that vague prompts produce vague results. So before I typed anything, I wrote down exactly what I wanted. Then I typed it all in one go:

"I want to build a website called Synaptic Stars Retro Universe. It needs to be a hub for my retro games — right now there are two: Barney's BinDaRoids (an asteroids-style blaster) and Barney's Bins: The Rubbish Run (an endless runner). I want it to look like a retro arcade cabinet — neon colours, retro pixel fonts, maybe a scanline effect — but feel modern and clean. I want: a logo for Synaptic Stars, a games section showing both games with links to play them, a blog section linking to my Misadventures in Vibing blog, an about section explaining what the site is, and a footer. It needs to be fully accessible. Build me the whole thing."

That's specific. It names both games. It describes the aesthetic. It lists every section I wanted. It asks for accessibility from the start. All of that matters — as you're about to see.

💡
Lesson 1: Every vague word in your prompt becomes a decision the AI makes for you. Sometimes you'll like the decision. Sometimes you'll get a unicorn. (More on that shortly.)

Entry 1 — Seven Files, One Session

What came back was seven files. In one go. Over two thousand lines of code. For a website I'd described in a paragraph.

  • index.html — the main landing page, fully structured with every section I asked for
  • css/style.css — a complete retro neon stylesheet, 786 lines, with accessibility support built in: skip links, focus-visible styles, reduced-motion support, high-contrast mode
  • js/sounds.js — a Web Audio API sound system, off by default, with a toggle button in the corner
  • js/main.js — the wiring for hover sounds, click sounds, and a CRT scanline overlay effect
  • blog/index.html — the blog listing page
  • blog/post-001.html — the first blog post
  • css/blog.css — separate styles for the blog, 562 lines

The neon colours were right. The fonts — Press Start 2P for the pixel headings, Orbitron for everything else — were exactly what I'd imagined. The scanline overlay was subtle and cool. The sound toggle worked. The accessibility was solid because I'd asked for it in the first prompt.

The logo, however, was a brain with a unicorn.

I want to be clear: I did not ask for a brain with a unicorn. I asked for a Synaptic Stars logo that felt retro and cool. The brain I could follow the logic on — synaptic, neurons, brains. The unicorn arrived without explanation and has never been fully explained.

Also: both game links pointed to netlify.app addresses that did not exist. I hadn't given the actual GitHub Pages URLs, so the AI guessed. The guesses were wrong.

Two problems. Logo and links. Both fixable.

💡
Lesson 2: The first build is a starting point, not a finished product. Expect to iterate. The goal of the first prompt is to get something real on screen that you can react to — not to be done.

Entry 2 — Getting It Live on GitHub Pages

Before fixing anything, I needed the site to actually be somewhere. GitHub, where I was hosting everything, has a feature called GitHub Pages that turns a repository into a live website. Free. No servers. No hosting fees.

The catch: it expects your files to be in a folder called docs/. My files were in a folder called synaptic-stars/, which is a sensible name but is not the name GitHub Pages is looking for.

"I want to deploy this to GitHub Pages. It needs the files to be in a docs/ folder to work. Can you set that up?"

This created a mirrored docs/ folder with all the same files. Now I had two copies — synaptic-stars/ as the working source and docs/ as the live deployed version. Every update needs to go in both. It's a slightly clunky setup, but it works, and understanding why it's structured this way made me feel briefly like I understood something about web infrastructure.

💡
Lesson 3: Tell the AI where you're deploying from the very start. "I'm hosting on GitHub Pages" upfront means the folder structure gets built correctly the first time. Don't retrofit it.

Entry 3 — The Logo Redesign and the URL Problem (Part One)

With the site live, I turned back to the two problems: the logo and the broken links.

For the logo, I wanted something more intentional than a brain-unicorn situation. More cartoon. More like something you'd see on the side of an actual arcade cabinet:

"The logo needs redesigning — can you make it more cartoon-style, like something you'd see on an arcade cabinet? Keep the brain and unicorn but make them look more fun, more retro, with bold outlines, cartoon eyes, rosy cheeks. More personality."

What came back was significantly better. The brain got filled cartoon styling — gradient fill, bubbly bumps, cartoon eyes with iris and pupil and a little shine dot, rosy cheeks, a big smile. The unicorn got chubby rounded proportions, a rainbow layered mane, a gold spiral horn with a glow effect, cartoon lashes, a cute smile.

I also gave the actual game URLs at the same time:

"Also the game links are wrong — BinDaRoids should link to synapticstar101.github.io/Barneys-BinDaRoids/ and The Rubbish Run should link to synapticstar101.github.io/Barneys-bins/"

The links were updated across every file — game tiles, footer navigation, blog pages, all of it.

Except The Rubbish Run URL was still wrong.

⚠
What actually happened: the AI had replaced the original placeholder (barneysrubbish.netlify.app) with a guessed GitHub Pages URL: synapticstar101.github.io/Barneys-Bins-The-Rubbish-Run/. That repository is called Barneys-bins, not Barneys-Bins-The-Rubbish-Run. Two different names. One working link, one 404.
💡
Lesson 4: Always paste your actual URLs. Don't describe them, don't paraphrase them — copy and paste the real link from your browser. Repository names are exact and case-sensitive. One wrong character means a broken link.

Entry 4 — The URL Problem (Part Two)

I noticed the Rubbish Run link was still broken and sent a correction:

"The Rubbish Run link is still wrong. The correct URL is synapticstar101.github.io/Barneys-bins/ — please update it everywhere."

This fixed it across all four HTML files — both the synaptic-stars/ source and the docs/ deployed copies, index pages and blog pages. Everything updated in one pass.

Two rounds of URL fixes for one link. This is the kind of thing that sounds embarrassing in retrospect and is actually just completely normal. You give imprecise information, you get imprecise results, you correct it. That's the process.

💡
Lesson 5: When you ask for something to be updated "everywhere," check everywhere. Not just the page you're looking at — the footer, the blog, the nav. If there are multiple copies of a file (like docs/ and synaptic-stars/), check both.

Entry 5 — The Game Previews and the Unicorn Cleanup

The game tiles on the homepage had basic placeholder graphics — they worked but they didn't look like the games. I wanted someone landing on the site to immediately recognise what each game felt like from the thumbnail.

I also noticed the blog was listing the games in the wrong order. And there was a stray SVG element from the logo work that had leaked into the page and shouldn't have been there.

"The game preview images need updating. For BinDaRoids it should look like a proper arcade title screen — dark space background, cyan neon text glowing, INSERT COIN button, controls hints at the bottom. For The Rubbish Run it should match the actual game menu — dark navy background, gold BARNEY'S BINS title, white THE RUBBISH RUN subtitle, orange dividers, green PRESS ENTER prompt. Also fix the game order in the blog post so BinDaRoids comes first, then The Rubbish Run. And there's a leftover unicorn element that doesn't belong — remove it."

All of it landed correctly. The BinDaRoids tile now looks like its actual title screen — cyan glow, dark space, that arcade cabinet feel. The Rubbish Run tile matches the real game menu — gold text, navy background, the green start prompt. The blog order was corrected. The stray element was removed.

This is the detail pass. After the structure is right and the links work, you go back through and make everything look and feel like itself. It takes multiple rounds of "that's not quite right" to get there. That's normal.

💡
Lesson 6: Bundle related fixes together. One message with three things to fix is more efficient than three separate messages — and it means the AI can check for knock-on effects between the changes at the same time.

Entry 6 — The Logo Gets a Third Life: The Robot

I kept coming back to the logo. The cartoon brain-unicorn was better than version one. But it still wasn't right. It didn't feel like Synaptic Stars. It felt like something that had been designed by a committee that couldn't agree on what Synaptic Stars was.

I wanted something memorable. Something with a clear personality. Something that, if you saw it on a t-shirt, you'd know immediately what it was.

I scrapped the brain and the unicorn entirely and wrote the most detailed creative brief of the whole project:

"I want to completely replace the logo. Forget the brain and unicorn. I want a friendly retro robot mascot — think 1950s sci-fi toy robot, full body. Give it star-shaped glowing eyes in yellow, cone antennae on either side of the head with glowing purple balls on the tips, and a fizzing zigzag lightning bolt crackling between the two antenna balls. Rosy circular cheeks. A big friendly smile. Dials and a small control panel on the chest. Arms, legs, feet — the whole robot. Colour scheme: blue metallic for the body, purple for the antenna balls, neon glow effects on the eyes and lightning bolt. Scatter some sparkles around him. It should look like it belongs in a neon arcade circa 1983. Keep the SYNAPTIC STARS RETRO UNIVERSE text alongside it."

What came back was the robot you can see at the top of this site right now.

Star eyes. Purple glowing antenna balls. A wild yellow zigzag crackling between them. A forehead vent panel with little grid lines. Rosy cheeks. A wide smile. Arm clamps. A chest with dials and a mini control panel. Sparkle dots scattered around him. An afterglow shadow underneath his feet so he looks like he's glowing onto the floor.

Everything I described — and then details I didn't think to ask for. That's the thing about a precise brief: it gives the AI enough to work with that it can fill in the gaps well, rather than filling them in randomly.

The brain and the unicorn are gone. The robot remains.

💡
Lesson 7: Don't settle for version one of anything visual. Get something on screen, look at it, then describe what's wrong or what it should be instead. Iteration is the process — not the failure.
💡
Lesson 8: The more specific your visual brief, the better the result. "Make it look retro" is not a brief. "1950s toy robot, star-shaped glowing eyes, cone antennae with purple balls, fizzing lightning bolt between them, blue metallic, rosy cheeks, full body, sparkles" — that's a brief.

What Building a Website Taught Me About Vibe Coding

The games had taught me the basics. The website taught me the refinements.

Accessibility is free if you ask for it upfront.

This site has skip links, ARIA labels, keyboard navigation, reduced-motion support, and high-contrast mode — not because I went back and added them, but because I asked for "fully accessible" in the first prompt. Retrofitting accessibility is hard. Asking for it on day one is a single word.

Two folders means two places to update.

The synaptic-stars/ source and the docs/ deployed copy need to stay in sync. Every time I make a change to one, I have to make it to the other. It's not complicated, but it's easy to forget — and if you forget, the live site and the source drift apart. Ask the AI to update both every time.

Check every link before you call something done.

I had one broken link that took two rounds of fixing because I didn't verify the actual URL carefully enough the first time. Paste the URL from your browser. Click every link after it's been updated. Don't assume it's right.

The logo will iterate. That's fine.

The robot on the homepage is version three of the logo. Version one was a brain with an unexplained unicorn. Version two was a better cartoon brain with a better cartoon unicorn. Version three is what it should always have been. None of this was wasted — each version revealed what wasn't quite right, which is exactly how you get to what is.

SVG graphics built entirely in code are remarkable.

The robot at the top of this page — the star eyes, the purple antenna balls, the crackling lightning bolt, the rosy cheeks, the dials on his chest, the sparkle dots around him — is built entirely from SVG coordinates, shapes, gradients, and filters. No image files. No Photoshop. Hundreds of lines of geometry that resolve into a cheerful robot with a glow effect. I didn't write any of it. I described what I wanted clearly enough that it arrived.

— Barney, Non-Dev, Accidental Web Designer
